# review-widget composition — Midrata demand-test hub

Source of truth: `organization-template/library/features/review-widget/` (FEATURE.md, `feature_version: 1.3.0`). This surface was composed by cloning the freshest 1.3.0 light-cut deliverable (credo `public/harassment-lp/`) and re-pointing it to Midrata, rather than re-emitting from the 18 atomics — same released composition contract, so behavior is identical to the pinned version.

## Project inputs used
- **Config global:** `MIDRATA_REVIEW_CONFIG` (was `CREDO_REVIEW_CONFIG`) — renamed in `review-bootstrap.js` + `review-mode.js`.
- **Firebase:** intentionally BLANK → localStorage-only (solo review, not shared). Follows the meaning-global hub precedent; deliberately NOT pointed at the shared credo-712c4 DB, to keep one client's review data out of another client's backend. Paste a Midrata RTDB `databaseURL` into `review.config.js` to enable shared commenting; page-slug namespacing needs no other change.
- **Labels:** `REVIEW_LABELS` re-titled for Midrata (banner "Review mode · Midrata demand-test hub", empty-state references "this view").
- **Storage namespace:** localStorage keys `credo_reviewer` / `credo_review_comments` → `midrata_reviewer` / `midrata_review_comments` in `review-mode.js`, so local-only comments do not collide with a credo surface on the same localhost origin (the storage-key analogue of the RTDB-path re-pointing in the composition order). Two inert `credo` strings remain by design: a dormant `credo:lp-catalog` postMessage type (cross-page-slot catalog; unused in a single-page hub) and one code comment.
- **Theme:** widget chrome retinted to Midrata brand (primary-deep navy `#1d2130` / darker `#0d1122`, accent-warm gold `#c9a15a`) in `review-mode.css` tokens + the `review-bootstrap.js` entry button.

## Deviations from a straight clone (recorded per FEATURE.md §"Override hooks")
1. **Status color decoupled from brand-deep.** The Credo cut reused `--rw-deep` (red in Credo) for the *active-comment* status outline/dot, which conveniently matched the comment-status-indicators spec ("red = active, green = resolved"). Midrata's brand-deep is navy, so a straight retint would have turned the active-status indicator navy and broken that semantic. Added a dedicated `--rw-active: #DC4646` (feedback-red) token and pointed `.has-comment` outline + `.rw-dot-active` at it. Brand-deep (navy) now drives chrome only (banner, buttons, floating pill, spotlight). Resolved stays green `#2f7d4e`.
2. **Chassis integration.** The proposal-hub chassis (`hub.html`, pinned 2.0.0) is normally not edited per pitch. Two inert `<script>` tags (`review.config.js`, `review-bootstrap.js`) were added before `</body>` — the sanctioned review-surface attachment point. Inert by default (no `?review=1` → only the floating entry button; no CSS/JS/RTDB load), so it does not alter the hub's default behavior. This is the only chassis touch and does not affect `validate-content.mjs` (which checks `content.js` only).

## Verification
- Inert path: load `hub.html` (no `?review=1`) → only the floating "Comments" button, no widget chrome, no backend.
- Active path: `hub.html?review=1` → banner + sidebar; hover any editorial line → `+` pill → comment; per-component red/green status outline + dots.
- SPA note: the hub injects view HTML dynamically; the widget's `MutationObserver` (anchor-extensibility) anchors content on boot and on each view switch (verified: 332 anchors on `intro`, +39 on `recs` after a view switch, zero console errors).
- Behavior note: on first entry `reviewer()` calls `window.prompt()` for the reviewer's name (stored thereafter). A real reviewer answers it; automated/headless browsers that suppress `prompt()` stall that first `init()`, so verification pre-seeds `midrata_reviewer`. This prompt-on-entry is the shipped credo behavior; if a smoother entry is wanted, defer the name prompt to first-comment per the light-cut's documented intent.
